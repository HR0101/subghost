## 3. 基本的な用語集

| 用語 | 説明 |
|------|------|
| **リポジトリ (repository)** | プロジェクトのファイルや変更履歴を保存する場所．． |
| **リモートリポジトリ** | GitHub 上にあるリポジトリ．チームで共有する「原本」． |
| **ローカルリポジトリ** | 自分のPC上にあるリポジトリ．ここで作業する． |
| **ブランチ (branch)** | 独立した作業ライン．本流（`main`）に影響を与えずに開発できる． |
| **クローン (clone)** | リモートリポジトリのコピーをローカルに作成すること． |
| **プル (pull)** | リモートから最新の変更を取得してローカルに反映すること． |
| **フェッチ (fetch)** | リモートの変更情報を取得するだけ（マージはしない）． |
| **アッド (add)** | 変更をコミット対象として「ステージング」すること． |
| **コミット (commit)** | ステージングした変更をローカルリポジトリに記録すること． |
| **プッシュ (push)** | ローカルの変更をリモートリポジトリに送信すること． |
| **マージ (merge)** | あるブランチの変更を別のブランチに統合すること． |
| **プルリクエスト (PR)** | 変更内容のレビュー・マージを依頼する仕組み． |
| **コンフリクト (conflict)** | 同じ箇所を複数人が編集したときに発生する競合． |
| **HEAD** | 現在チェックアウトしているブランチの最新コミットを指すポインタ． |
| **origin** | デフォルトのリモートリポジトリの名前（別名でも可）． |

## 4. 初期セットアップ（最初の1回だけ）

### Git のインストール確認
```bash
git --version
```

### ユーザー情報の設定
コミットには誰が変更したかの情報が記録されるため，必ず設定します．
```bash
git config --global user.name "あなたの名前"
git config --global user.email "your-email@example.com"
```

### 設定の確認
```bash
git config --list
```

---

## 5. 基本的な Git コマンド

###  リポジトリのクローン
GitHub上のリポジトリをローカルにコピーします．**初回のみ**実行．
```bash
git clone <リポジトリのURL>
```

実行後，リポジトリ名のフォルダが作成されるので，そこに移動します．
```bash
cd <リポジトリ名>
```

---

###  現在の状態を確認
```bash
git status        # 変更されたファイルやステージング状態を確認
git branch        # ブランチ一覧と現在地を確認
git log --oneline # コミット履歴を簡潔に表示
```

>  **困ったらまず `git status`**．今どんな状態かを教えてくれます．

---

###  ブランチの作成と切り替え
**作業前に必ずブランチを作りましょう**．`main` で直接作業するのは厳禁です．

```bash
git checkout -b <新しいブランチ名>
```



---

###  変更のステージング
編集したファイルを，コミット対象として登録します．
```bash
git add <ファイル名>     # 特定のファイル
git add .                # 全ての変更
git add -p               # 変更箇所を1つずつ確認しながら追加（おすすめ）
```

---

###  変更のコミット
ステージングした変更をローカルリポジトリに保存します．
```bash
git commit -m "コミットメッセージ"
```

#### 良いコミットメッセージの書き方
**Prefix（接頭辞）を付ける**と分かりやすくなります．
```
feat: ログイン機能を追加
fix: ヘッダーの表示崩れを修正
docs: READMEに使い方を追記
style: インデントを修正
refactor: ユーザー認証処理を整理
test: ログイン機能のテストを追加
chore: 依存パッケージを更新
```

**悪い例**: `修正`, `更新`, `aaa`
**良い例**: `fix: ログインボタンが押せないバグを修正`

---

###  リモートリポジトリへのプッシュ
ローカルの変更を GitHub に送信します．
```bash
git push origin <ブランチ名>
```

初回プッシュ時は以下でも可（次回以降は `git push` だけでOK）:
```bash
git push -u origin <ブランチ名>
```

---

###  リモートからの取得と反映
他のメンバーの変更を取り込みます．

> ⚠️ **重要**: `git pull` する前には，**必ず `main` ブランチに戻ってから**行いましょう．

```bash
git checkout main
git pull origin main
```

その後，作業ブランチに戻って `main` の変更を取り込みます．
```bash
git checkout <作業ブランチ>
git merge main
```

---

###  ブランチのマージ
```bash
git checkout <マージ先のブランチ名>
git merge <マージ元のブランチ名>
```

>  実際のチーム開発では，コマンドでマージするより **GitHub上のプルリクエスト経由でマージ**するのが一般的です．

---



## 6. プルリクエスト（PR）の作成手順

1. ブランチを GitHub にプッシュする．
2. GitHub のリポジトリページにアクセス．
3. **「Pull requests」タブ** → **「New pull request」** をクリック．
4. **base**（マージ先：通常は `main`）と **compare**（マージ元：作業ブランチ）を選択．
5. タイトルと説明を記入．
   - **何を変更したか**
   - **なぜ変更したか**
   - **どうテストしたか**
   - 関連 Issue 番号があれば記載（例: `Closes #12`）
6. レビュアーを指定して「Create pull request」をクリック．



## 7. やってはいけないこと（初心者がやりがちなミス）

|  NG行動 | 理由 |
|----------|------|
| `main` ブランチで直接作業する | 他の人に影響する．必ずブランチを切る． |
| `git push --force` を気軽に使う | 他人のコミット履歴を破壊する可能性がある． |
| 巨大な変更を1つのコミットにする | レビューしづらく，原因特定も困難になる． |
| パスワードやAPIキーをコミットする | 公開リポジトリだと情報漏洩．`.gitignore` を活用． |
| コミットメッセージを「修正」だけにする | 後から見て何の修正か分からない． |
| pull せずに作業を始める | コンフリクトの原因になる． |

## 8. `.gitignore` ファイルについて

Git で管理したくないファイル（ログ，環境変数，ビルド成果物など）を指定します．

### `.gitignore` の例

# 環境変数
.env

# ログファイル
*.log

# OS固有のファイル
.DS_Store
Thumbs.db

# 依存パッケージ
node_modules/
__pycache__/

# IDEの設定
.vscode/
.idea/

## 9. チーム開発で大切な心構え

1. **こまめにコミット・プッシュする** — 作業の見える化につながる．
2. **`git pull` を習慣にする** — コンフリクトを未然に防ぐ．
3. **わからないときはすぐ聞く** — 強制プッシュで取り返しがつかなくなる前に．
4. **コミットメッセージは未来の自分とチームへの手紙** — 丁寧に書く．
5. **レビューは批判ではなく学びの場** — 指摘されたら感謝して直す．
