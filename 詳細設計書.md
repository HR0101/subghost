# Ghostty補助ノッチAIアシスタント 詳細計画書

## 1. プロジェクト概要

### 1.1 目的
MacBookのノッチ領域を活用し、Ghosttyで実行中のAI CLI（Claude Code / Codex CLI / Antigravity）を
補助的に監視・操作するmacOSネイティブアプリケーションを開発する。

作業の主戦場はあくまでGhosttyであり、本アプリは以下の補助的役割に徹する。

- AIの実行状態（考え中/完了/エラー）をノッチに可視化
- 応答完了時の通知と要約チラ見せ
- Ghosttyに切り替えずにノッチからクイックプロンプトを投入
- 頻用プロンプト/スニペットの即時挿入

### 1.2 設計思想
- **主役はGhostty、ノッチは脇役**：フル対話はGhostty側で完結
- **読み取り＋軽い注入に徹する**：責務を絞って壊れにくくする
- **ヒューリスティックで十分**：厳密なパースより実用性を優先

### 1.3 対象環境
- macOS（ノッチ搭載機：MacBook Pro 14/16インチ、MacBook Air等）
- Ghostty ターミナルエミュレータ
- tmux（セッション橋渡しに使用）

## 4. 機能仕様

### 4.1 役割1：AI状態インジケーター（優先度：最高）
- ノッチ左右余白に状態アイコンを常時表示
- 状態：`idle`（待機） / `thinking`（生成中） / `completed`（完了） / `error`（エラー）
- 色分け：グレー / 青パルス / 緑 / 赤
- 判定ロジック：
  - pane末尾テキストが伸長中 → `thinking`
  - 一定時間（例：1.5秒）変化停止＋プロンプト記号出現 → `completed`
  - エラーパターン検出 → `error`

### 4.2 役割2：完了通知＋要約チラ見せ（優先度：最高）
- 応答完了時にノッチが下方向に展開（Dynamic Island風アニメーション）
- 最終応答の冒頭3〜4行を表示
- クリックでGhosttyを最前面化（`NSWorkspace` でアクティベート）
- 数秒後に自動で折りたたみ

### 4.3 役割3：クイックプロンプト送信（優先度：高・必須）
- グローバルショートカット（デフォルト：`⌥Space`）でノッチ入力欄を展開
- `TextField` にプロンプト入力 → Enterで `send-keys` 送信
- 送信後は入力欄を閉じ、状態インジケーターが `thinking` に遷移
- 送信先セッションが複数ある場合はドロップダウンで選択

### 4.4 役割4：プロンプト履歴/スニペット（優先度：中）
- 頻用プロンプトを登録（例：「テストも書いて」「日本語で」「リファクタして」）
- ノッチ展開時にスニペット一覧をチップ表示、クリックで挿入
- 送信履歴を保存し、上矢印キーで呼び出し

## 15. 追補：承認リクエスト / 質問への回答（Approve / Ask）

CLIが選択肢を出して停止している間、従来の状態機械は「出力が静止した」とみなして `completed` と誤判定していました.本追補ではこれを独立した状態として扱い、ノッチから回答できるようにします.

### 15.1 追加した状態

| 状態 | 意味 | 表示色 |
|---|---|---|
| `awaitingApproval` | 権限リクエストの承認待ち | 橙(点滅) |
| `awaitingAnswer` | エージェントからの質問待ち | 黄(点滅) |

`AIState.needsUserResponse` が真の間はユーザーの回答がないと処理が進みません.

### 15.2 検出ロジック（`ChoicePrompt`）

`capture-pane` のテキストから、枠線・スピナー・ANSIを除去した末尾40行を走査します.

1. **番号付きメニュー**：`^(?:[❯>▶●•*]\s*)?(\d{1,2})[.)]\s+(.+)$` に合致する行を、最後の合致行から上へ連番で収集する
2. 収集結果が**2件以上かつ1から始まる連番**のときのみ選択肢とみなす（誤検出防止）
3. ブロック直上の非ノイズ行を問いかけ本文、さらに上の最大3行を文脈として取る
4. `(y/n)` 形式は末尾5行以内にある場合のみ、Yes/No の2択として扱う
5. 問いかけ本文が `CLIProfile.approvalPattern` に合致すれば `approval`、しなければ `question` に分類する

### 15.3 状態機械への組み込み

`StateDetector.ingest` の**冒頭**（他のどの判定よりも先）で選択肢を検出します.

- 選択肢あり かつ 前回と内容が異なる → `becameAwaitingChoice`
- 選択肢あり かつ 前回と同一 → `none`（再通知しない）
- 選択肢が消えた → `choiceResolved` → `thinking` へ復帰

ノッチから回答した直後は画面がまだ更新されていないため、`answeredSuppressionInterval`（既定4秒）の間だけ同じ選択肢を無視します.これを過ぎても同じ画面なら、キーが届かなかったとみなして再通知します.

### 15.4 回答の送信

`TmuxClient.sendChoice` が選択肢のキーを `send-keys -l` でリテラル送信します.番号選択メニューは番号キーだけで確定するため Enter は送らず、`(y/n)` 形式のみ `needsEnter` で Enter を続けて送ります.

**安全設計**：Subghostが自動で回答することは一切ありません.送信はユーザーの明示的な操作（クリック / キー入力 / 通知ボタン）に限られます.`Esc` はCLIへ何も送らずノッチを閉じるだけです.

### 15.5 UI（`NotchMode.choice`）

- 回答待ちの間は自動折りたたみを行わない
- 数字キー / `⏎`（先頭を選択）/ クリックで回答、`Esc` で閉じるのみ
- 「はい」系は緑、「いいえ」系は赤のバッジで色分け
- 閉じた問い合わせは `dismissedChoices` に記録し、内容が同じ限り再展開しない
- 入力モード中は割り込まず、入力を閉じた時点で改めて展開する

### 15.6 通知

`interruptionLevel = .timeSensitive` を設定し、集中モード中でも届くようにします.はい/いいえが揃っている場合はバナーに「承認」「拒否」ボタンを出し、そこからも回答できます.

### 15.7 サウンドアラート（`SoundAlerts`）

音源ファイルを持たず、`AVAudioEngine` で矩形波を合成して8bit風のモチーフを鳴らします.

| イベント | モチーフ |
|---|---|
| 応答完了 | G5 → C6（上行） |
| エラー | G#4 → D#4（下行） |
| 承認待ち | A5・A5・D6（3連） |
| 質問 | E5 → B5 |

音の切れ目は3msのフェードでクリックノイズを防ぎます.オーディオデバイスが使えない場合は `NSSound.beep()` にフォールバックします.独自音が有効な間は、通知音と二重に鳴らないよう `UNNotificationContent.sound` を `nil` にします.

## 16. 追補：ターミナルへの移動（Jump）

対応ターミナルを Ghostty とターミナル.app の2種に定め、「AI CLIが動いているタブ」まで正確に移動できるようにします.

### 16.1 対応ターミナル

| ターミナル | bundle ID | タブ単位の移動 |
|---|---|---|
| Ghostty | `com.mitchellh.ghostty` | 不可（AppleScript非対応）→ アプリの前面化のみ |
| ターミナル.app | `com.apple.Terminal` | **可**（AppleScriptで `tty` の一致するタブを選択） |

### 16.2 ホストするターミナルの特定

「どのターミナルのどのタブか」は、tmuxのクライアントttyを起点に判定します.

1. `tmux list-clients -t <session> -F '#{client_tty}'` でアタッチ中クライアントのtty（例 `/dev/ttys003`）を得る
2. `ps -t ttys003 -o pid=` でそのtty上のプロセスを列挙する
3. `ps -A -o pid=,ppid=` で作った親子表を使い、各pidの祖先をたどる
4. 祖先に起動中ターミナルのpidが現れたら、それがホストのターミナル

デタッチ中のセッションはクライアントttyが存在しないため、アプリの前面化にフォールバックします.祖先の追跡は最大40段で打ち切り、親子関係が循環していても停止します.

実測例：`ttys003` 上のシェル → `claude` → `login` → `Ghostty(pid 687)` と正しくたどれることを確認済み.

### 16.3 ターミナル.app のタブ選択

`NSAppleScript` で全ウインドウ・全タブを走査し、`tty` プロパティが一致するタブを `selected` にしてウインドウを前面化します.

**セキュリティ**：ttyの文字列はAppleScriptのソースへ埋め込むため、`isValidTTY` で `/dev/` 始まり・英数字と`/`のみ・32文字以内に限定してから使います.これによりスクリプト注入を防ぎます.

**権限**：Apple Events の送信には自動化の許可が必要です.`INFOPLIST_KEY_NSAppleEventsUsageDescription` に利用目的を記載しており、初回実行時に許可ダイアログが出ます.拒否された場合は `executeAndReturnError` がエラーを返すため、アプリの前面化にフォールバックします.

### 16.4 設定

「移動先のターミナル」は既定で**自動判定**です.16.2で特定できなかった場合にのみ、この設定値（Ghostty / ターミナル.app）が使われます.

### 16.5 命名の整理

Ghostty専用ではなくなったため、以下を改名しました.

- `GhosttyActivator` → `TerminalActivator`
- `GhosttySessionWatcher` → `SessionWatcher`
- `AppCoordinator.openGhostty()` → `jumpToTerminal()`

## 17. 追補：ゼロコンフィグ検出（エイリアスの廃止）

従来は「`ai-` で始まる名前のtmuxセッション」だけを監視対象としており、ユーザーに `~/.zshrc` へのエイリアス登録を強いていました.本追補で、実行中プロセスからAI CLIを直接検出する方式へ移行します.

### 17.1 検出の原理

`ps -A -o pid=,tty=,comm=` の出力を解析します.

- **`comm` はカーネルが保持する実行ファイル名**であり、CLIが自身のプロセス名を書き換えていても正しい値が得られます.
  Claude Codeは `tmux` の `#{pane_current_command}` にバージョン文字列（例 `2.1.215`）を出すため、そちらは判定に使えません.
- **制御端末を持たないプロセス（tty が `??`）は除外**します.これによりGUIアプリが同梱するバイナリ（例: ChatGPT.appの `/Applications/ChatGPT.app/Contents/Resources/codex`）を誤検出しません.
- 実行ファイル名は `CLIProfile.executableNames` との**完全一致**で判定します.`claude-helper` のような別物を拾わないためです.

### 17.2 tmuxとの突き合わせ

`tmux list-panes -a -F "#{pane_tty}|#{session_name}:#{window_index}.#{pane_index}"` で全ペインを列挙し、pty → ペイン宛先の対応表を作ります.検出したプロセスのttyがこの表にあれば、そのペインが `capture-pane` / `send-keys` の宛先になります.

セッション名を一切見ないため、**命名規則もエイリアスも不要**になりました.またペイン単位で宛先を指定するため、分割ペインでも正しいペインを操作します.

### 17.3 同一性の軸

`SessionInfo` の識別子を tmuxセッション名から **tty** へ変更しました.ttyはターミナルの1タブ／1ペインに対応するため、tmuxの有無に関わらず安定した識別子になります.同じtty上でCLIが起動し直された場合（pidの変化）は、状態機械ごと作り直します.

### 17.4 監視できないセッションの扱い

tmuxの外で動くCLIは画面を読む手段がないため、`SessionInfo.isMonitorable` を `false` とし、以下のように扱います.

- 一覧には表示する（検出はできている）
- Jump は可能（ttyが分かるため、むしろtmux経由より直接的）
- 状態は判定せず「監視不可（tmux外）」と表示する
- 送信先ドロップダウン・`Tab` 循環からは除外し、送信を試みた場合は `SessionError.notMonitorable` を返す

### 17.5 実機での確認

本マシンで検証したところ、tmuxを使わず起動されたClaude Codeが3つ動作していました（ttys000 / ttys003 / ttys004）.旧方式ではこれらを1つも検出できていませんでした.新方式の解析結果は実際の `ps` 出力と一致することを確認済みです.

## 18. 調査記録：tmuxなしで画面を読めるか

tmux依存を完全に外せるかを調査しました.

### 18.1 ターミナル.app：**可能**（AppleScript）

Accessibility APIを使うまでもなく、AppleScriptがタブの画面テキストをそのまま返します.

| プロパティ | 内容 |
|---|---|
| `contents of <tab>` | 現在表示中の画面テキスト |
| `history of <tab>` | スクロールバックを含む履歴 |

実際に取得できることを確認済みです.`capture-pane` の完全な代替になります.

注意点：`contents` はAppleScript組み込みの参照演算子と名前が衝突するため、`tell <tab> to get its contents` の形で呼ぶ必要があります.

送信側は `do script "..." in <tab>` が使えますが、**末尾に改行が付く**ため、承認メニューの「番号キーのみ（Enter不要）」を再現できません.単一キー送信にはSystem Eventsの `keystroke`（アクセシビリティ権限が必要）が要ります.

### 18.2 Ghostty：**部分的にのみ可能（実用不可）**

GhosttyはAppleScript非対応のため、Accessibility APIが唯一の候補です.調査の結果、画面テキスト自体は取得できることを確認しました.

```
AXWindow/AXGroup/AXGroup/AXScrollArea/AXTextArea → AXValue（4241文字を取得）
```

しかし、常駐監視に使うには致命的な制約が3つあります.

| 制約 | 実測 | 影響 |
|---|---|---|
| `AXTextArea` は**前面タブの1つだけ** | タブ4つに対し要素は1個 | 裏のタブは読めない＝バックグラウンド監視が成立しない |
| `AXValue` が**書き込み不可** | `AXUIElementIsAttributeSettable` = false | 送信・承認ができない（CGEvent等での擬似キー入力が別途必要） |
| `AXWindows` が**空** | 0件（`AXFocusedWindow` のみ取得可） | 複数ウインドウを列挙できない |

加えてAXにはtty属性がなく、タブのタイトルしか手がかりがないため、**検出したプロセス（tty）とタブの対応付けができません**.

Subghostの中心的な価値は「別の作業をしている間に複数セッションを見張る」ことなので、前面タブしか読めない時点でGhosttyについてはtmuxの代替になりません.

### 18.3 結論

| ターミナル | tmuxなしでの監視 | 根拠 |
|---|---|---|
| ターミナル.app | **可能** | AppleScriptが全タブの `contents` と `tty` を返す |
| Ghostty | **不可** | AXは前面タブのみ・書込不可・tty対応付け不能 |

したがってtmux依存を一律に外すことはできません.採りうる方針は「ターミナル.appのみtmux不要にする（非対称仕様）」か「両者ともtmuxを前提に統一する（現状維持）」のいずれかです.

### 18.4 調査コードの所在

`scratchpad/axprobe*.swift`（調査用、製品コードには含めない）.

なお初回の調査では `AXWindows` のみを走査して0件だったため「サンドボックスの制約で不明」と誤って結論づけました.実際には `AXFocusedWindow` から辿れば取得でき、走査対象の誤りでした.

## 19. 追補：フック方式（Claude Code / Codex）

画面のテキストを解析する代わりに、Claude Codeのフックからイベントを直接受け取る方式を追加しました.tmuxを介さないため、Ghosttyでも全機能が動作します.

### 19.1 方式の比較

| | tmux方式 | フック方式 |
|---|---|---|
| 状態の取得 | `capture-pane` の正規表現解析 | イベントを直接受信 |
| 承認の実現 | `send-keys` でキー送出 | フックの戻り値で allow/deny |
| tmux | 必須 | 不要 |
| Ghostty | 動作する（tmux経由） | 動作する |
| CLIのUI変更 | 壊れうる | 影響なし |
| ポーリング | 必要 | 不要 |
| 対応CLI | 画面パターンを書けば対応可 | フックを持つCLIのみ |

両者は併存し、フックが届いているセッションでは画面解析を行いません（`isHookConnected` で切り替え）.

### 19.2 経路

```
Claude Code ──フック実行──▶ ~/.subghost/bin/subghost-bridge
                                  │ curl --unix-socket
                                  ▼
                          ~/.subghost/run/subghost.sock
                                  │
                                  ▼
                          Subghost（HookServer）
```

ブリッジは専用バイナリではなくシェルスクリプトです.`curl` がUnixドメインソケット上のHTTPを話せるため、追加のビルドターゲットが要りません.

**ソケットのパスをホーム直下の `~/.subghost/` にしている理由**：`sockaddr_un.sun_path` はmacOSで104バイト制限があり、`~/Library/Application Support/…` では超える恐れがあるためです.実際に検証中 `AF_UNIX path too long` を踏んで判明しました.

### 19.3 CLIを壊さないための設計

Subghostが停止・削除されてもClaude Codeが影響を受けないことを最優先にしています.

- フックのコマンドは `[ -x "$SCRIPT" ] && "$SCRIPT" claude; exit 0` の形で、**スクリプトが無ければ何もせず正常終了**する
- スクリプト自身も `[ -S "$SOCK" ] || exit 0` でソケット不在なら即終了する
- `curl` が失敗しても `|| exit 0`
- Subghost側が解釈できない入力を受けたら、空オブジェクト `{}` を返して**CLI本来の確認画面に委ねる**
- アプリ終了時は待たせている接続をすべて解放する（CLIを停止させたままにしない）

### 19.4 settings.json の書き換え

- 実行前に確認ダイアログを出し、**日時付きバックアップ**を作成する
- 既存の設定・他ツールのフックは保持し、Subghostの項目のみ追記する
- 自分の項目は**コマンド末尾のシェルコメント `# subghost-bridge`** で識別する.
  設置先パスに目印が含まれることに依存すると、パスを変えた際に解除できなくなるため（テストで発見）
- 解除時は自分の項目だけを取り除き、そのイベントが空になればキーごと削除して元の状態に戻す

### 19.5 承認の流れ

1. Claude Codeが `PermissionRequest` フックを実行し、応答があるまで停止する
2. ブリッジがイベントJSONをソケットへ送り、接続を保持したまま待つ
3. Subghostがノッチに「許可 / 拒否」を表示する
4. ユーザーの選択を `permissionDecision` として返す
5. ブリッジがそれを標準出力へ出し、Claude Codeが判定として読む

**キー送出が一切不要**なため、ターミナルの種類に依存しません.

タイムアウトは1800秒で、超過した場合は素通しとなりCLI本来の確認画面に戻ります.

### 19.6 未検証の点

`permissionDecision` を返すJSONの形は、Claude Codeのフック出力仕様に関する現時点の理解に基づくもので、**実物との突き合わせは未実施**です.形が違っていた場合はClaude Code側が指示を無視し、通常どおりユーザーに確認するだけなので、**壊れるのではなく「効かない」に留まります**.

検証済みなのは以下です（モックサーバによる実地確認）.

- ソケット不在時にフックが終了コード0で抜けること
- 接続を保持してブロックできること（承認待ちの再現）
- 判定JSONがフックの標準出力に出ること
- 非ブロックイベントが即座に `{}` を返すこと

## 20. 追補：Codexのフック対応

Codexにもフック機構があることを確認し、対応しました.

### 20.1 Claude Codeとの差異

| | Claude Code | Codex |
|---|---|---|
| 定義ファイル | `~/.claude/settings.json` の `hooks` キー | `~/.codex/hooks.json`（専用ファイル） |
| JSONの構造 | `hooks` → イベント名 → matcher配列 | **同じ** |
| イベント | 9種 | **6種**（`Notification` / `PreToolUse` / `SessionEnd` / `StopFailure` が無い） |
| matcher | `"*"` | `""`（空文字が全一致） |
| 信頼の承認 | 不要 | **必要** |

JSONの構造が同じため、`addHooks` / `removeHooks` は共通のまま `HookTarget` で差分を吸収しています.

Codex固有のイベントとして `SubagentStop` があります.これは**サブエージェントの終了**であり、親エージェントはまだ作業中です.完了と誤って扱わないよう `thinking` へ倒しています.

### 20.2 Codexの信頼機構

Codexは `config.toml` の `[hooks.state]` に各フックのsha256を記録し、信頼済みのフックのみを実行します.`--dangerously-bypass-hook-trust` という迂回フラグも存在します.

**Subghostはこのハッシュを自分で書き込みません.** 信頼はユーザーが与えるべきセキュリティ上の判断であり、アプリが勝手に「信頼済み」を偽装することは、この機構を無効化する行為だからです.設定画面でその旨を案内し、Codex起動時の正規の承認フローに委ねます.

### 20.3 イベント名の表記ゆれ

CLIによってイベント名やキー名の表記が異なる可能性があるため、解釈を緩めています.

- イベント名: `PermissionRequest` / `permission_request` / `PERMISSIONREQUEST` を同一視する
- キー名: `hook_event_name` / `hookEventName` / `hook_event` / `event_name` / `event` を順に探す

いずれも見つからなければ解釈せず素通しにします.

### 20.4 Antigravityについて

**対応を見送りました.**

検証環境に `antigravity` コマンドが存在せず（`which antigravity` が not found）、フック機構の有無・設定ファイルの位置・イベント名・ペイロード形式のいずれも確認できませんでした.

推測で設定ファイルを書き込むことは、ユーザーの環境を壊す危険があるうえ、動作確認もできません.Antigravityは引き続きtmux方式で監視します.CLIが導入された環境で `~/.antigravity` 配下などに設定機構が確認できれば、`HookTarget` に1ケース追加するだけで対応できる構造にしてあります.

## 21. 追補：表示先ディスプレイの選択

複数モニタ環境で、ノッチUIをどの画面に出すかを設定できるようにしました.

### 21.1 背景

従来は `NSScreen.screens.first { $0.safeAreaInsets.top > 0 }` でノッチ搭載画面を無条件に選んでいました.しかし「内蔵ディスプレイにノッチがあり、外部モニタをメインにして作業する」構成では、ユーザーが見ていない画面にUIが出てしまいます.検証環境がまさにこの構成でした.

| 画面 | ノッチ | メイン |
|---|---|---|
| DELL S2725DC (2560×1440) | なし | ✓ |
| Built-in Retina Display (1512×982) | ✓ | — |

### 21.2 設定値

`DisplayPreference` の3状態をUserDefaultsのキー `notchDisplay` に保存します.

| 値 | 意味 |
|---|---|
| `""` | 自動（ノッチ搭載画面 → メイン → 先頭） |
| `"main"` | 常にメインディスプレイ |
| （UUID文字列） | 特定のディスプレイ |

### 21.3 ディスプレイの識別

`CGDisplayCreateUUIDFromDisplayID` で得られるディスプレイUUIDを識別子にしています.`CGDirectDisplayID` は再接続で変わりうるため、設定の保存先には使えません.UUIDが取得できない場合のみ表示名で代用します.

実機での取得例:

```
DELL S2725DC          → 6C3BE65B-370B-4E66-B3E3-19683D40D4C1
Built-in Retina Display → 37D8832A-2D66-02CA-B9F7-8F30A301B230
```

### 21.4 フォールバック

指定した画面が接続されていない場合（外部モニタを外した等）は、自動と同じ規則に倒します.設定値自体は保持するため、**再接続すると元の画面に戻ります**.

### 21.5 テスト可能な形への分離

選択ロジックは `NSScreen` に依存すると単体テストできないため、必要な属性だけを持つ `ScreenDescriptor`（id / 表示名 / ノッチの有無 / メインか）に対する純粋関数 `DisplaySelector.select` として切り出しました.`NSScreen` からの変換は `descriptor` プロパティが担う薄い層です.

これにより「外部モニタを外したときのフォールバック」「メイン指定がノッチ搭載画面より優先される」といった分岐を、実機の画面構成に依存せず検証できます.

### 21.6 再配置

以下のタイミングで寸法と位置を取り直します.

- 設定変更時（`AppCoordinator.reloadDisplayPlacement`）
- 画面構成の変化時（`NSApplication.didChangeScreenParametersNotification`）

ノッチの無い画面を選んだ場合は、従来からある擬似ノッチ（メニューバー直下に同形のバーを描画）で表示します.

## 22. 追補：起動時に既存セッションの状態を引き継ぐ

Subghostを後から起動した場合でも、すでに動いているCLIの現在状態を拾えるようにしました.

### 22.1 問題

プロセス検出（追補17）により、起動中のCLIは以前から一覧に出ていました.しかし**状態は必ず「待機」から始まっていました**.

`StateDetector.ingest` は画面の差分（出力が伸びた等）から状態を判断する設計のため、初回は基準値を置くだけで `.none` を返します.選択待ちの検出もこの基準値保存より後ろにあり、初回は到達しません.

その結果、**Subghost起動前から承認待ちで止まっているセッションを取りこぼしていました**.ユーザーが画面を見に行くまで、CLIが止まっていることに気づけません.

### 22.2 対応

初めて見るセッションには `adoptCurrentState` を使い、差分ではなく**現在の画面そのもの**から状態を推定します.

| 画面の内容 | 引き継ぐ状態 |
|---|---|
| 選択肢が表示されている | 承認待ち / 質問中（ノッチを開く） |
| busyパターンに一致 | 生成中 |
| それ以外 | 待機 |

### 22.3 完了を意図的に返さない理由

`adoptCurrentState` は `.becameCompleted` を返しません.

Subghost起動前に終わっていた応答について「完了しました」と通知や音を出すのは、ユーザーにとって誤報だからです.すでに読み終えているかもしれない古い結果を、今起きたことのように知らせるべきではありません.

一方で**承認待ちは必ず拾います**.こちらは過去の出来事ではなく、今この瞬間もCLIが停止して操作を待っている状態だからです.

### 22.4 起動順序の修正

`AppCoordinator.start()` で、監視の開始より先にノッチパネルを生成するよう順序を入れ替えました.1回目のポーリングが承認待ちを検出した際、パネル未生成だとノッチを開けないためです.

### 22.5 フック方式の場合

フックはイベント発生時にのみ発火するため、起動前から待機しているセッションの状態は取得できません.同じセッションがtmux配下にもあれば、上記の引き継ぎで状態が埋まります.

将来的には `~/.claude/projects/*/<session>.jsonl` の末尾を読むことで、フックのみの環境でも現在状態を復元できる可能性があります（未実装）.

## 23. 不具合修正：ノッチ非搭載画面でメニューバーの下にずれる

### 23.1 症状

外部モニタ（ノッチ非搭載）を表示先にすると、UIがメニューバーの位置ではなく**その下**に表示される.

### 23.2 原因

`NotchMetrics.compute()` が、ノッチの有無で `topY`（パネル上端のy座標）の基準を変えていた.

| 画面 | 使っていた値 |
|---|---|
| ノッチ搭載 | `screen.frame.maxY`（画面の絶対上端） |
| ノッチ非搭載 | `screen.visibleFrame.maxY`（**メニューバーを除いた**領域の上端） |

`visibleFrame` はメニューバーぶんを差し引いた矩形なので、その上端に置くと必ずメニューバーの高さだけ下にずれる.

実測値（検証環境）:

| 画面 | `frame.maxY` | `visibleFrame.maxY` | 差 |
|---|---|---|---|
| DELL S2725DC（外部） | 1440 | 1410 | **30** |
| Built-in Retina Display | 0 | −32 | 32 |

DELLでは上端1440に対して1410へ配置しており、ずれ幅30ptはメニューバーの高さと完全に一致する.

### 23.3 修正

- `topY` は**ノッチの有無によらず `frame.maxY`**（画面の絶対上端）を使う
- `topInset`（バーの高さ）は決め打ちの `34` をやめ、`frame.maxY - visibleFrame.maxY` で**実測**する

決め打ちの34は実測値（外部30 / 内蔵32）のどちらとも一致しておらず、これも誤りだった.`NSStatusBar.system.thickness` は22を返すため、これも使えない.

メニューバーを測れない場合（自動非表示、副画面にメニューバーが無い構成）は既定値24ptに倒す.

### 23.4 テスト可能な形への分離

`NSScreen` に依存すると単体テストできないため、寸法計算を純粋関数 `NotchMetrics.make(frame:visibleFrame:safeAreaTop:auxiliaryWidths:)` に切り出した.検証環境の実測値をそのままテストケースにしてある.

これにより「ノッチの有無によらず上端の基準が同じであること」を回帰テストとして固定できる.今回の不具合はまさにこの不変条件が崩れていたものだった.

## 24. 追補：メニューバーが出ている時だけ表示する

### 24.1 経緯

サブモニタでアプリを全画面表示すると、メニューバーが隠れているにもかかわらずノッチUIだけが残り、アプリの内容に重なってしまっていた.

### 24.2 使えなかった判定方法

`NSScreen.visibleFrame` は全画面表示中も変化しませんでした（実測）.

| 状態 | DELLの `frame.maxY` | `visibleFrame.maxY` |
|---|---|---|
| 通常時 | 1440 | 1410 |
| 全画面時 | 1440 | 1410（**変わらない**） |

`visibleFrame` は「デスクトップ空間のレイアウト」を表す値であり、全画面スペースの状態を反映しません.したがって寸法計算には使えても、表示/非表示の判定には使えません.

### 24.3 判定方法（当初の誤りと修正）

**当初の実装（誤り）**: 「対象画面にメニューバーが出ているか」だけで判定していました.

これは**メニューバーがフォーカスされている画面にしか描画されない**ことを見落としていました.その結果、別の画面で全画面アプリを使っているだけで、対象画面のノッチまで消えてしまいました.

**修正後**: 2つの条件を組み合わせます.

```
表示する = メニューバーが出ている || 全画面アプリに覆われていない
```

隠すのは「**対象画面自身が**全画面アプリに覆われていて、かつメニューバーも出ていない」場合だけです.

| 状況 | メニューバー | 覆われている | 結果 |
|---|---|---|---|
| 通常のデスクトップ | あり | なし | 表示 |
| 対象画面が全画面 | なし | あり | **隠す** |
| 別の画面が全画面（対象画面は通常） | なし | なし | **表示**（修正点） |
| 全画面中にメニューバーを呼び出した | あり | あり | 表示 |

### 24.4 メニューバーの検出

`CGWindowListCopyWindowInfo` でウインドウ一覧を取り、メニューバーのウインドウが対象画面の上端にあるかを見ます.

全画面時の実測値:

```
layer=24  bounds=x=207 y=1440 w=1512 h=33   ← 内蔵ディスプレイのメニューバー
layer=0   bounds=x=0   y=38   w=2560 h=1402 ← DELLの全画面アプリ
```

判定条件:

1. `kCGWindowLayer == 24`（`kCGMainMenuWindowLevel`）
2. 高さが10〜60ptの範囲（画面全体を覆うレイヤー24のオーバーレイを除外）
3. 上端が対象画面の上端と一致（±2pt）
4. 横方向の重なりが画面幅の半分を超える（別画面のメニューバーを拾わない）

**ウインドウ名（`kCGWindowName`）は参照していません.** 名前の取得には画面収録の権限が必要になるためです.

条件2は実測で必要性が判明しました.スクリーンショットUIがレイヤー24かつ画面全体（2560×1440）で存在しており、高さで除外しないと誤検出します.

### 24.5 全画面の検出

通常のウインドウはメニューバーより上に配置できないため、「画面の上端に接していて幅を9割以上覆う通常ウインドウ（レイヤー0）」を全画面スペースとみなします.

実測では、全画面のアプリが `y=0`（画面上端）から始まっており、通常の最大化ウインドウ（メニューバーの下から始まる）と区別できます.

### 24.6 使えなかった判定方法

`NSScreen.visibleFrame` は全画面表示中も変化しませんでした（実測）.

| 状態 | DELLの `frame.maxY` | `visibleFrame.maxY` |
|---|---|---|
| 通常時 | 1440 | 1410 |
| 全画面時 | 1440 | 1410（**変わらない**） |

`visibleFrame` は「デスクトップ空間のレイアウト」を表す値であり、全画面スペースの状態を反映しません.寸法計算には使えても、表示/非表示の判定には使えません.

### 24.7 例外：操作が必要なときは隠さない

`displayMode == .compact` のときだけ隠します.承認待ち・質問・入力中は、全画面アプリの内容に重なってでも表示します.

応答しないとCLIが停止したままになるため、「見えないので気づけない」状態を作らないことを優先しました.

### 24.8 追従

0.4秒間隔で監視し、状態が変わったときだけ `orderOut` / `orderFrontRegardless` を呼びます.表示先ディスプレイを切り替えた直後は、その画面の状態で判定し直します.

異常系（ウインドウ一覧が取得できない、画面IDが取れない）では**表示する**側に倒します.消えたまま戻らないより、出ているほうが安全なためです.

## 25. 不具合修正：パネルが全画面アプリの下に潜り込む

### 25.1 症状

- 全画面アプリの上にノッチUIが表示されない
- 表示されている場合でもクリックが反応しない

### 25.2 原因

パネルのウインドウレベルが、コードで指定した26ではなく**3（`.floating`）になっていた**.

動作中のアプリを `CGWindowListCopyWindowInfo` で調べて判明した.

```
layer=3 onscreen=false x=1141 y=0 w=278 h=30
```

原因は初期化の**順序**である.

```swift
panel.level = ...26...        // ここで26を指定しても
panel.isFloatingPanel = true  // これが level を .floating(3) に上書きする
```

`isFloatingPanel = true` は副作用としてウインドウレベルを `.floating` に設定する.この順序では指定した26が3に潰される.

レベル3はメニューバー(24)・ステータスバー(25)・全画面アプリのいずれよりも下になるため、

- 全画面スペースでは他のウインドウの下に隠れて見えない
- 上に別のウインドウがあるとクリックがそちらへ吸われる

という2つの症状が同時に出ていた.

検証コードで順序依存を確認した.

```
levelを先に設定（不具合時）    : 指定=26 → 実際=3
isFloatingPanelを先に設定（修正）: 指定=26 → 実際=26
```

### 25.3 修正

設定を `NotchPanelController.configure(_:)` に切り出し、`level` を必ず `isFloatingPanel` の後に設定するようにした.

修正後、実際に動作しているアプリで `layer=26` になることを確認済み.

### 25.4 再発防止

この種の順序依存はコードを読んでも気づきにくく、静かに再発する.そのため設定処理を関数として切り出し、以下を単体テストで固定した.

- レベルがメニューバー・ステータスバーより上であること
- レベルが `.floating`(3) に上書きされていないこと（本不具合そのもの）
- 全画面スペースに参加する `collectionBehavior` であること
- マウス入力を受け取る設定であること

## 26. 不具合修正：フックのイベントがセッションに紐づかない

### 26.1 前提の訂正

「Claude Codeはフック設定を起動時に読むため、設置前から動いているセッションではフックが効かない」と説明していたが、**これは誤りだった**.

実測したところ、設置前から動いているセッションでもフックは発火していた.設定は動的に読み直されており、**CLIの再起動は不要**である.

### 26.2 本当の原因

フック自体は発火していたが、Subghost側でどのセッションのイベントか特定できず、すべて捨てられていた.

原因は2つある.

**(1) 親プロセスに制御端末が無い**

ブリッジは `ps -o tty= -p $PPID` でttyを取得していたが、実測すると `??` が返っていた.

```
自分のpid=47005 ppid=47003
  祖先: 47003  3514 ??       /bin/sh    ← 親。制御端末を持たない
  祖先:  3514 96778 ttys003  claude     ← CLI本体。ttyはここにある
```

フックは `/bin/sh -c '...'` 経由で起動され、この中間シェルは制御端末を持たない.親を1段見るだけでは特定できない.

**(2) ttyの表記が食い違っていた**

仮にttyが取れても、ブリッジは `ttys003`、セッション側は `/dev/ttys003` の形式で保持しており、文字列比較が永久に一致しなかった.

この2つが重なり、突き合わせに失敗 → `respondPassthrough` でイベント破棄 → `hookSessionID` も設定されない → **以降のイベントも永久に紐づかない**という状態になっていた.

### 26.3 修正

**ブリッジ側**: 祖先を最大10段たどり、最初に制御端末を持つプロセス（＝CLI本体）を探す.そのpidとttyを両方ヘッダで送る.

```
X-Subghost-Tty: /dev/ttys003
X-Subghost-Pid: 3514
```

**サーバ側**: ttyを正規化（`ttys003` → `/dev/ttys003`）し、突き合わせを確実な順に多段化した.

1. pid（最も確実）
2. tty
3. CLI側セッションID
4. 作業ディレクトリ（同名が1つだけの場合）

### 26.4 修正の配布

ブリッジはシェルスクリプトであり、アプリの更新だけでは置き換わらない.そのため**起動時に、登録済みであればスクリプトを最新内容へ書き直す**ようにした.ユーザーが再登録しなくてもスクリプト側の修正が反映される.

### 26.5 回帰テスト

- ttyの表記ゆれを正規化すること（本不具合そのもの）
- 取得できなかった場合にnilへ倒すこと
- ブリッジが祖先探索を行い、pidと `/dev/` 付きttyを送ること
