# File: NOTCH_AI_ASSISTANT_PLAN.md

# Ghostty補助ノッチAIアシスタント 詳細計画書

## 1. プロジェクト概要

### 1.1 目的
MacBookのノッチ領域を活用し、Ghosttyで実行中のAI CLI（Claude Code / Codex CLI / Antigravity）を
補助的に監視・操作するmacOSネイティブアプリケーションを開発する。

作業の主戦場はあくまでGhosttyであり、本アプリは以下の補助的役割に徹する。

- AIの実行状態（考え中/完了/エラー）をノッチに可視化
- 応答完了時の通知と要約チラ見せ
- Ghosttyに切り替えずにノッチからクイックプロンプトを投入
- 頻用プロンプト/スニペットの即時挿入

### 1.2 設計思想
- **主役はGhostty、ノッチは脇役**：フル対話はGhostty側で完結
- **読み取り＋軽い注入に徹する**：責務を絞って壊れにくくする
- **ヒューリスティックで十分**：厳密なパースより実用性を優先

### 1.3 対象環境
- macOS（ノッチ搭載機：MacBook Pro 14/16インチ、MacBook Air等）
- Ghostty ターミナルエミュレータ
- tmux（セッション橋渡しに使用）

## 2. 対応AI CLI

クイックプロンプト送信が必須要件のため、全CLIをtmuxセッション内で実行する方式に確定する。

| CLI | コマンド例 | 特徴 | 状態判定の注意点 |
|-----|-----------|------|-----------------|
| Claude Code | `claude` | 対話モード、ストリーミング出力 | スピナー/思考中表示の除去が必要 |
| Codex CLI | `codex` | 対話モード | プロンプト記号の識別が必要 |
| Antigravity | `antigravity` | 対話/エージェント型 | ツール実行中の長時間待機に対応 |

### 2.1 CLIごとの抽象化
各CLIの出力パターンを吸収するため、`CLIProfile` として設定を外部化する。

- プロンプト待ち受け記号のパターン
- 「考え中」を示すスピナー/文言のパターン
- エラー出力のパターン
- 応答区切りの判定ルール

## 3. アーキテクチャ

### 3.1 全体構成

```
[Ghostty]
    │
    │ 起動
    ▼
[tmux session: ai-<cli名>]
    │
    ▼
[AI CLI プロセス] (claude / codex / antigravity)
    ▲         │
    │送信      │出力キャプチャ
    │         │
[ノッチアプリ (SwiftUI + AppKit)]
    ├─ SessionWatcher   : capture-pane で状態監視
    ├─ PromptSender     : send-keys でプロンプト注入
    ├─ NotchOverlay     : ノッチUI描画
    ├─ NotificationMgr  : 完了通知
    └─ SnippetStore     : スニペット管理
```

### 3.2 橋渡し方式：tmux
- **監視**：`tmux capture-pane -p -t <session>` を定期ポーリング
- **注入**：`tmux send-keys -t <session> "prompt" Enter`
- **セッション検出**：`tmux list-sessions` で対象セッションを列挙

### 3.3 コンポーネント責務

| コンポーネント | 責務 |
|---------------|------|
| `GhosttySessionWatcher` | pane出力の監視、状態遷移の判定 |
| `PromptSender` | tmuxへのプロンプト送信 |
| `CLIProfile` | CLIごとの出力パターン定義 |
| `NotchOverlayWindow` | ノッチ直下のNSPanelオーバーレイ |
| `NotchViewModel` | UI状態の保持・更新 |
| `NotificationManager` | 完了/エラー通知 |
| `SnippetStore` | スニペット/履歴の永続化 |
| `SessionManager` | 監視対象セッションの選択・切替 |

## 4. 機能仕様

### 4.1 役割1：AI状態インジケーター（優先度：最高）
- ノッチ左右余白に状態アイコンを常時表示
- 状態：`idle`（待機） / `thinking`（生成中） / `completed`（完了） / `error`（エラー）
- 色分け：グレー / 青パルス / 緑 / 赤
- 判定ロジック：
  - pane末尾テキストが伸長中 → `thinking`
  - 一定時間（例：1.5秒）変化停止＋プロンプト記号出現 → `completed`
  - エラーパターン検出 → `error`

### 4.2 役割2：完了通知＋要約チラ見せ（優先度：最高）
- 応答完了時にノッチが下方向に展開（Dynamic Island風アニメーション）
- 最終応答の冒頭3〜4行を表示
- クリックでGhosttyを最前面化（`NSWorkspace` でアクティベート）
- 数秒後に自動で折りたたみ

### 4.3 役割3：クイックプロンプト送信（優先度：高・必須）
- グローバルショートカット（デフォルト：`⌥Space`）でノッチ入力欄を展開
- `TextField` にプロンプト入力 → Enterで `send-keys` 送信
- 送信後は入力欄を閉じ、状態インジケーターが `thinking` に遷移
- 送信先セッションが複数ある場合はドロップダウンで選択

### 4.4 役割4：プロンプト履歴/スニペット（優先度：中）
- 頻用プロンプトを登録（例：「テストも書いて」「日本語で」「リファクタして」）
- ノッチ展開時にスニペット一覧をチップ表示、クリックで挿入
- 送信履歴を保存し、上矢印キーで呼び出し

## 5. 状態判定ロジック詳細

### 5.1 ポーリング
- 間隔：0.8秒（負荷と反応速度のバランス）
- `capture-pane -p -e`（エスケープ含む）または `-p`（プレーン）を選択可能に

### 5.2 状態遷移表

| 現在状態 | 条件 | 次状態 |
|---------|------|--------|
| idle | 出力が伸長 | thinking |
| thinking | 出力が伸長継続 | thinking |
| thinking | N秒変化なし＋プロンプト記号 | completed |
| thinking | エラーパターン検出 | error |
| completed | 新規プロンプト送信 | thinking |
| completed | ユーザー確認/一定時間経過 | idle |
| error | 出力が伸長 | thinking |

### 5.3 ANSI/スピナー処理
- `capture-pane -p` でプレーンテキスト取得を基本とする
- 残存する制御文字・スピナー文字（例：`⠋⠙⠹`）は正規表現で除去
- CLIプロファイル側で除去パターンを定義

## 6. UI/UX仕様

### 6.1 ノッチオーバーレイ
- `NSPanel`（`.nonactivatingPanel`, `.borderless`）
- `NSScreen.safeAreaInsets` / `auxiliaryTopLeftArea` / `auxiliaryTopRightArea` でノッチ位置を算出
- 常時レイヤー：`.statusBar` 以上
- 全Space表示：`collectionBehavior = [.canJoinAllSpaces, .stationary]`

### 6.2 表示モード
- **コンパクト**：状態アイコンのみ（ノッチ両サイド）
- **展開（通知）**：応答チラ見せ、自動折りたたみ
- **展開（入力）**：プロンプト入力欄＋スニペットチップ

### 6.3 アニメーション
- 展開/折りたたみは `withAnimation(.spring)` で滑らかに
- `thinking` 中はアイコンをパルスアニメーション

### 6.4 インタラクション
- ホバー：コンパクト→軽く展開でプレビュー
- クリック：Ghostty最前面化
- グローバルショートカット：入力モード展開

## 9. セキュリティ・安全設計

- **シェル経由を避ける**：`Process` で引数配列を使い、文字列連結でのコマンド実行をしない
- **プロンプト注入の無害化**：send-keysに渡す前に制御文字をサニタイズ
- **サンドボックス**：tmux実行に必要な範囲でエンタイトルメントを設定（必要に応じ非サンドボックス配布）
- **権限**：グローバルショートカットのためアクセシビリティ権限が必要 → 初回に案内

## 10. 起動フロー / セットアップ

### 10.1 ユーザー側セットアップ
シェル設定にエイリアスを登録する。

```
# ~/.zshrc
alias aiclaude='tmux new-session -A -s ai-claude claude'
alias aicodex='tmux new-session -A -s ai-codex codex'
alias aiantigravity='tmux new-session -A -s ai-antigravity antigravity'
```

Ghosttyで上記エイリアスを実行するだけで、本アプリが対象セッションを自動検出する。

### 10.2 アプリ側
1. 起動時に `tmux list-sessions` で `ai-*` セッションを検出
2. 検出したセッションを監視対象に登録
3. 複数ある場合はアクティブセッションを選択可能に

## 12. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| CLIの出力形式が更新される | 状態判定が壊れる | プロファイルを外部設定化し容易に修正可能に |
| tmux非導入ユーザー | 使えない | セットアップガイド提供、将来はログ監視方式も検討 |
| ポーリング負荷 | バッテリー消費 | 間隔調整、idle時は間隔を延ばす |
| ノッチ非搭載機 | UI崩れ | メニューバー表示にフォールバック |
| アクセシビリティ権限拒否 | ショートカット不可 | 権限案内、メニューからの操作を代替提供 |
| 誤った完了判定 | 通知の誤発火 | 判定の閾値を調整可能に、CLIごとに最適化 |
